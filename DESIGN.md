## Problem Statement

Le "Fossé de Réalité" : un programme CNC est généré dans un environnement numérique
parfait. Le monde physique (brides mal placées, décalage d'origine, usure d'outil)
ne coopère pas. Le crash catastrophique est le cas extrême. Le problème quotidien
est différent et bien plus fréquent :

- **Babysitting de première pièce** : à chaque nouveau programme, l'opérateur doit
  rester à côté de la machine, main sur le bouton E-stop, incapable de prendre sa
  pause ou de gérer ses autres machines.
- **Machine idle pendant les pauses** : 20-30 minutes perdues par première pièce
  parce que l'opérateur ne peut pas quitter la machine en sécurité.
- **Dégradation silencieuse d'outil** : les opérateurs juniors ne reconnaissent pas
  encore le son d'une mauvaise coupe. L'outil se dégrade sans alerte, parfois pendant
  des heures.
- **Blocage de nuit** : opérateur de nuit, programmeur absent, erreur de code
  non-corrigeable → travail arrêté jusqu'au lendemain. Pièce urgente non livrée.

---

## Demand Evidence

Expérience directe, vécue en tant qu'opérateur CNC :

- A géré 3-4 machines simultanément, opérations de nuit, sans programmeur disponible.
- A posé des brides à des endroits de collision non détectés par le programme.
  Sans vigilance constante, crash et casse d'outil.
- A dû arrêter une machine et attendre le lendemain matin pour une correction de code.
- Pause de midi impossible sur une première pièce : "la machine tournait, je ne pouvais
  pas partir."
- Observation : les opérateurs seniors "entendent" le problème. Les juniors non. Ce
  savoir-faire prend des années à acquérir. Pendant ces années, les dommages
  silencieux s'accumulent.
- Contexte démographique : peu de jeunes entrent dans le métier. Les shops peinent
  à recruter des juniors capables. Un outil qui transfère le savoir-faire senior
  résout un problème de pénurie, pas juste de sécurité.

---

## Status Quo (ce que les opérateurs font aujourd'hui)

1. **Main sur E-stop** : l'humain est le capteur de sécurité. Coût = attention totale.
2. **Surveillance obligatoire sur premières pièces** : impossible de gérer d'autres
   machines ou de prendre une pause.
3. **Oreille du senior** : savoir-faire tacite, non-transférable, long à acquérir.
4. **Attendre le programmeur** : si problème détecté la nuit, travail bloqué.

Solutions existantes : Marposs (capteurs de force intégrés, enterprise, cher),
Neuron Soundware (SaaS acoustique, enterprise). Aucune solution accessible pour
les petites shops de 4-5 machines. Aucune ne continue le job — toutes s'arrêtent.

---

## Utilisateur Cible et Wedge le Plus Étroit

**Utilisateur principal** : opérateur CNC junior ou intermédiaire dans une PME
manufacturière de 4-10 personnes, gérant 2-4 machines, souvent en sous-effectif.

**Le moment critique** : la première exécution d'un nouveau programme sur une
nouvelle pièce. C'est là que 100% de la valeur est concentrée.

**Le wedge** : "Ta machine continue de tourner pendant ta pause." Pas "on évite
les crashes" — ça c'est le bénéfice secondaire. La proposition primaire c'est
l'autonomie machine sur les premières pièces.

## Constraints

- Projet hybride : démonstration académique (portfolio, stage) ET fondations
  pour une startup potentielle.
- Budget étudiant : hardware accessible (Raspberry Pi, webcam, micro directionnel).
- Timeline académique : doit être démontrable dans un délai raisonnable.
- Marché cible : shops qui ne peuvent pas justifier $250k-350k pour une nouvelle
  machine. Retrofit play sur machines fonctionnelles.

---

## Premises

1. Le problème réel = "Reality Gap" programmer/opérateur. Le crash est la
   conséquence extrême. La perte de productivité quotidienne (babysitting,
   pauses bloquées, machine idle) est la douleur fréquente et mesurable.

2. Marché cible = PME 4-10 personnes, machines fonctionnelles, retrofit play.
   Pas les grandes usines (déjà servies par Marposs, Neuron Soundware).

3. Pour le projet académique, le pilier visuel est plus impressionnant en démo.
   Pour le wedge commercial, l'acoustique est plus accessible techniquement.
   Les deux apparaissent dans l'architecture, à des fidelités différentes.

4. Le positionnement c'est "autonomie machine" pas "système de sécurité".
   La différenciation clé = on ne s'arrête pas, on contourne.

## Approches Considérées

### Approche A : Acoustic-First MVP
Résumé : Pilier sonore uniquement. FFT sur Raspberry Pi + micro directionnel.
         Détecte mauvaise coupe, alerte opérateur. Vision en V2.
Effort  : S
Risque  : Faible
Pour    : Facile à shipper, mesurable, bonne histoire business
Contre  : Moins spectaculaire en démo académique. Pas de path replanning.
Complétude : 6/10

### Approche B : Vision-First Demo
Résumé : Simulateur 2D/3D + webcam. Outil détecté, obstacles détectés,
         algorithme d'évitement. Demo visuelle percutante. Acoustique absent.
Effort  : M
Risque  : Moyen
Pour    : Maximum "wow" pour recruteurs et profs.
Contre  : Moitié du produit manquante. Moins de valeur réelle immédiate.
Complétude : 7/10

### Approche C : Dual-Pillar Narrative ✓ CHOISIE
Résumé : Deux piliers, deux fidelités. Acoustique = implémentation réelle
         (FFT fonctionnel sur hardware). Vision = demo logicielle (simulateur
         2D + path replanning autour d'un obstacle). Un boîtier, une histoire
         cohérente, deux demos complémentaires.
Effort  : L
Risque  : Moyen (scope plus large mais piliers découplés)
Pour    : Raconte l'histoire complète. Demo spectaculaire + valeur réelle.
         Path replanning = différenciateur unique vs TOUS les concurrents.
Contre  : Plus de travail. Path replanning temps réel sur embedded = complexe.
Complétude : 9/10

## Approche Recommandée : Dual-Pillar Narrative

### Architecture

```
SENTINELLE CNC
├── Pilier Acoustique (implémentation réelle, V1 — machine bois ou fraiseuse légère)
│   ├── Micro USB ou MAX9814
│   ├── FFT glissante (fenêtre 512ms, overlap 50%, numpy/scipy)
│   ├── Baseline : moyenne des magnitudes FFT sur les 10 premières secondes de coupe
│   ├── Détection anomalie : amplitude crête ≥ baseline + 2σ sur fréquence fondamentale
│   │   Scope V1 : surcharge soudaine uniquement (pic brusque), PAS dégradation graduelle
│   ├── Séquence de réponse : alerte UI → attente 3s → si non acquittée → E-stop GPIO
│   └── Mécanisme faux positif : bouton mute 30s + compteur faux positifs logué
│
├── Pilier Visuel (simulateur laptop, V1 — PAS sur Raspberry Pi)
│   ├── Webcam standard (USB)
│   ├── Détection obstacles : masque HSV (objet couleur distinctive, ex. orange vif)
│   │   Fallback V1 : ArUco markers si conditions lumineuses variables
│   ├── Trajectoire outil : G-code simulé parsé depuis fichier .nc (pas curseur souris)
│   ├── Détection collision imminente : intersection bounding box outil/obstacle
│   ├── Path replanning A* : grille 50×50, recalcul < 200ms sur laptop standard
│   │   Déclenchement : à chaque obstacle détecté, fréquence max 5Hz
│   └── Affichage : trajectoire originale (rouge) + nouvelle trajectoire (verte)
│
└── Interface unifiée (UI pygame sur laptop, données acoustiques via WebSocket depuis Pi)
    ├── Pi → laptop : WebSocket (ws://pi.local:8765), JSON par événement
    │   Schéma : {"type": "acoustic", "severity": "warn|critical", "ts": <epoch_ms>}
    ├── Laptop → UI : {"type": "vision", "obstacle": true, "replanned": true, "ts": <epoch_ms>}
    ├── UI : fenêtre pygame unique, split-screen gauche (simulateur visuel) / droite (graphe FFT + alertes)
    ├── Conflit simultané : si acoustic.severity="critical" → overlay rouge prioritaire, stoppe simulateur
    └── Log JSON de session : fichier .jsonl local, tous les événements horodatés
```

**Note d'architecture V1/V2** :
- V1 : acoustique sur Pi, visuel sur laptop. Les deux affichent sur la même UI.
- V2 : path replanning physique sur CNC nécessite contrôleur GRBL (open-source, accès série).
  Sur Fanuc/Haas (contrôleurs fermés), l'injection G-code mid-programme est impossible ou payante.
  **Contrainte dure V2 : machine GRBL requise pour le path replanning physique.**

### Séquence de build recommandée

**Étape 1 — Simulateur 2D (Phase 1 du blueprint)**
- Environnement Python + pygame, tournant sur laptop
- Trajectoire outil : G-code simplifié parsé depuis un fichier .nc (lignes G0/G1 uniquement)
  → coordonnées converties en pixels pour l'affichage simulateur
- Obstacle : objet physique orange posé devant la webcam, détecté par masque HSV
- A* recalcule en temps réel dès qu'un obstacle intersecte la trajectoire prévue
- Livrable : vidéo de démo "obstacle détecté → trajectoire recalculée en vert"

**Étape 2 — Pilier acoustique réel**
- Raspberry Pi + micro USB ou MAX9814
- Script Python : capture audio → FFT → baseline → détection anomalie
- Test sur perceuse ou fraiseuse bois avant CNC réelle
- Livrable : alerte qui se déclenche quand on force l'outil

**Étape 3 — Intégration narrative**
- Pi tourne le script acoustique + serveur WebSocket (asyncio + websockets)
- Laptop tourne le simulateur visuel + client WebSocket (reçoit événements Pi)
- UI pygame split-screen : visuel à gauche, FFT + alertes à droite
- ~1-2 jours de travail spécifiquement pour l'IPC WebSocket + synchronisation UI
- Livrable : démo "copilote complet" sur une seule fenêtre, présentable en entretien

### Différenciateur clé : path replanning + l'opérateur reste dans la boucle

Tous les concurrents font : détection → arrêt d'urgence.
L'architecture SENTINELLE est conçue pour : détection → recalcul → **proposition à l'opérateur** → si confirmé → machine continue.

**Pourquoi "proposition" et pas "automatique" ?**
La modification autonome de G-code en cours de programme engage la responsabilité légale
du fournisseur du système (certification SIL/ISO requise, processus long et coûteux).
En gardant l'opérateur dans la boucle (il confirme la nouvelle trajectoire avant exécution),
SENTINELLE est un outil d'aide à la décision, pas un pilote automatique. La responsabilité
reste à l'opérateur. C'est aussi pourquoi les industriels enterprise n'ont pas fait ça —
pas faute d'idée, mais faute de volonté de gérer la certification.

**Flow V1 (simulateur) :**
1. Obstacle détecté (objet orange dans le champ webcam)
2. Simulation pause + affiche : trajectoire originale rouge, trajectoire alternative verte
3. Bouton "Confirmer" à l'écran → le point virtuel suit la trajectoire verte
4. Obstacle retiré → retour à la trajectoire originale proposée

**Flow V2 (machine GRBL physique) :**
1. Vision détecte collision imminente → signal E-stop d'urgence (GPIO ou série)
2. Système affiche nouvelle trajectoire proposée en G-code sur l'écran
3. Opérateur confirme → injection de la nouvelle séquence G-code via port série GRBL
4. Machine reprend sur la trajectoire corrigée

En V1, ce principe est démontré sur simulateur (G-code parsé + obstacle webcam →
A* recalcule → opérateur confirme → le point suit le nouveau chemin). Ça suffit pour
un recruteur ou prof technique. Sois transparent : "V1 est un simulateur, V2 intègre GRBL."
Ça montre que tu connais tes contraintes, pas que tu les caches.

### À quoi ressemble la démo V1 visuellement

La démo se passe entièrement sur laptop. Aucune CNC nécessaire.

```
ÉCRAN LAPTOP (pygame split-screen)
┌──────────────────────────┬─────────────────────────┐
│   SIMULATEUR VISUEL       │   PILIER ACOUSTIQUE      │
│                           │                          │
│  ·····●→→→→○ (rouge)      │  FFT live               │
│        ↓                  │  ▁▃▅▇▅▃▁  ← normal      │
│  ···●→→⌒→→→ (vert)        │                          │
│  [obstacle orange]        │  [ALERTE] si pic         │
│                           │                          │
│  [CONFIRMER] [ANNULER]    │  Faux positifs: 0        │
└──────────────────────────┴─────────────────────────┘
```

Séquence de démo (30 secondes) :
1. Tu lances → le point bleu suit le chemin G-code préchargé
2. Tu poses un objet orange dans le champ de la webcam
3. Le chemin devient rouge (bloqué), un chemin vert apparaît (contournement A*)
4. Tu cliques "Confirmer" → le point suit le vert et contourne l'obstacle
5. Tu retires l'objet → retour au chemin original proposé

La webcam pointe vers la surface réelle, l'obstacle est un objet physique, le reste
est entièrement virtuel sur écran. Pas de main dans le cadre. Pas de CNC nécessaire.

## Success Criteria

- **Démo acoustique** : outil forcé → alerte déclenchée en ≤ 3s. Mesuré par
  timestamp loggué entre événement audio et notification UI. Reproductible 3x de suite.
  Machine de test V1 : fraiseuse bois ou perceuse (PAS CNC métal — baseline différente).
- **Démo visuelle** : objet orange posé dans le champ de la webcam → trajectoire
  recalculée affichée à l'écran en ≤ 200ms. Ancienne trajectoire visible en rouge,
  nouvelle en vert. Reproductible en conditions de lumière normales.
- **Faux positifs** : taux < 1 fausse alerte par run de 5 minutes sur coupe normale.
  Compteur affiché dans l'UI pendant la démo.
- **Pitch cohérent** : un recruteur ou prof comprend la valeur en 90 secondes.
- **Fondations startup** : architecture découplée, README clair, code sur GitHub,
  story racontable à un investisseur.

## The Assignment

Cette semaine, avant d'écrire une ligne de code :

**Parle à 2-3 propriétaires de shops CNC ou professeurs de techniques de
fabrication. Pose-leur une seule question : "Sur vos premières pièces, combien
de minutes par jour vos opérateurs restent-ils bloqués à côté de la machine
sans pouvoir partir ?" Si la réponse est 20 minutes ou plus, tu as ta validation.
Si c'est zéro, creuse pourquoi — peut-être qu'ils ont déjà une solution, ou
peut-être qu'ils ont adapté leur workflow autrement.**

Ce n'est pas "go build it." C'est une conversation de 10 minutes qui vaut
3 mois de développement en termes de validation.
